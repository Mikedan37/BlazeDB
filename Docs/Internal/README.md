# Internal Docs

This area is for maintainer/internal documentation that should not be treated as first-time product documentation.

## Intended audience

- Maintainers and release owners
- Contributors working on CI/release/process alignment

## Not for first-time adopters

If you are evaluating or integrating BlazeDB, start with:

- `README.md`
- `Docs/README.md`
- `Docs/GettingStarted/README.md`
- `Docs/API/API_REFERENCE.md` (with support-state caveats)

## Typical content here

- Durable maintainer notes that are still operationally useful
- Internal runbooks and policy/support references that are not onboarding docs

## Temporary analysis artifacts policy

Temporary analysis/process artifacts can be created during release preparation, but once their durable conclusions are absorbed into canonical docs, they should be removed from tracked git to keep the OSS surface lean and authoritative.
